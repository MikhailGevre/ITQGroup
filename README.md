# API 

### Создать документ. 
<br>Создаёт документ в статусе DRAFT. Номер генерируется автоматически.

### Получить документы.<br>
   API умеет:<br>
   ● вернуть один документ вместе с историей,  
   ● вернуть список документов по списку id (пакетное получение)

### Отправить на согласование<br>
   Принимает список id (от 1 до 1000) и пытается перевести каждый документ DRAFT -> SUBMITTED.<br>
   Для каждого id возвращает результат: успешно / конфликт / не найдено.
   Обработка каждого документа атомарная, результаты - по каждому id.

### Утвердить<br>
   Принимает список id (от 1 до 1000) и пытается перевести каждый документ SUBMITTED -> APPROVED.<br>
   Для каждого id вернуть результат: успешно / конфликт / не найдено / ошибка
   регистрации в реестре.
   Обработка каждого документа атомарная, результаты - по каждому id.

### Поиск документов<br>
Фильтры: статус, автор, период дат. Возвращает список найденных документов. (по дате создания)

###  Проверка конкурентного утверждения
Отдельный API, который запускает несколько параллельных попыток
утвердить один и тот же документ (параметры threads и attempts) и возвращает
сводку:<br> сколько попыток прошло успешно, сколько завершилось конфликтом/ошибкой, какой финальный статус документа.<br>
Поведение: ровно одна попытка переводит документ в APPROVED и создает запись в реестре.<br>
Остальные попытки завершаются без изменения документа и без повторной записи в регистр.<br>


# Утилита + фоновая обработка
### Утилита генерации
Отдельный модуль, который читает из файла параметров число N и создаёт N документов через API сервиса.

### Фоновые процессы в сервисе
В сервисе работают два фоновых процесса в отдельных потоках.
Размер пачки задается в файле параметров параметром batchSize и используется в обработке.

### SUBMIT-worker<br>
Регулярно проверяет БД и отправляет документы со статусом DRAFT на
согласование пачками по batchSize через пакетное API.

### APPROVE-worker<br>
Регулярно проверяет БД и отправляет документы со статусом SUBMITTED на
утверждение пачками по batchSize через пакетное API.

# Логи
Логи создаются при первом запуске в корневой папке проекта logs. <br>
C уровнями логирования: info, debug, trace, error, warn<br>
Конфигурация логов происходит в папке resource/logback-spring.xml

# Ошибки и базовая валидация
Глобальный обработчик ошибок, который перехватывает исключения<br>
● Единый формат ошибок: код + сообщение.<br>
● “не найдено” для отсутствующего документа.<br>
● “конфликт/недопустимая операция” для недопустимого перехода статуса.<br>
● Проверка входных данных (пустые значения и т.п.) - через библиотеку Jakarta.
